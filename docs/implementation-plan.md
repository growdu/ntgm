# 逆天改命算命软件实现计划文档

## 1. 文档目标

本文档用于把现有产品、设计、技术和模块方案转成可执行实现计划。

目标是明确：

1. 先实现什么
2. 每一阶段要交付什么
3. 每一阶段的技术任务是什么
4. 每一阶段的验收标准是什么

---

## 2. 实现原则

1. 先做最小闭环，不先做大而全。
2. 先做结构化结果，不先做复杂文案。
3. 先做 Web + 后端主链路，再增强 Android / iOS。
4. 先做规则驱动画像，再评估更复杂模型。
5. 每阶段都要有可演示结果。

---

## 3. 总体目标

首个可用版本需要跑通这条主链路：

1. 用户建档
2. 生成初始八字结果
3. 进入持续问答
4. 生成画像 V1 / V2
5. 给出历史人物匹配
6. 给出建议结果
7. 保存成长档案

---

## 4. 阶段划分

建议按 6 个阶段推进：

1. Phase 0：工程初始化
2. Phase 1：数据层与基础服务
3. Phase 2：画像 MVP 闭环
4. Phase 3：Web MVP 界面
5. Phase 4：移动端基础接入
6. Phase 5：增强与上线准备

---

## 5. Phase 0：工程初始化

### 目标

让项目进入可开发状态。

### 任务

1. 初始化 monorepo 目录结构
2. 创建 `apps/web`
3. 创建 `apps/mobile`
4. 创建 `services/api`
5. 创建 `services/worker`
6. 创建 `packages/sdk`
7. 创建 `packages/domain`
8. 创建 `packages/validation`
9. 创建 `infra/compose`
10. 创建 `.env.example`

### 技术任务

1. 初始化 `pnpm-workspace.yaml`
2. 初始化 Web 工程
3. 初始化 Expo 工程
4. 初始化 FastAPI 工程
5. 初始化 Celery Worker 工程
6. 初始化 Docker Compose 基础环境

### 交付物

1. 可运行的 monorepo
2. 可启动的 Web / API / Worker
3. 本地 PostgreSQL / Redis / MinIO 开发环境

### 验收标准

1. 新开发者可按文档完成本地启动
2. Web 页面能打开
3. API 健康检查可访问
4. Worker 可消费测试任务

---

## 6. Phase 1：数据层与基础服务

### 目标

建立后续所有业务的基础数据和服务骨架。

### 任务

1. 设计并落地数据库表
2. 完成 ORM 模型
3. 完成 Alembic 初始迁移
4. 完成用户基础资料接口
5. 完成 intake records 写入链路
6. 完成 life events 写入链路
7. 完成对象存储上传链路
8. 完成任务状态模型

### 技术任务

1. 建 `users`
2. 建 `intake_records`
3. 建 `life_events`
4. 建 `image_assets`
5. 建 `profile_versions`
6. 建 `profile_change_logs`
7. 建 `match_results`
8. 建 `advice_plans`
9. 建任务状态表或状态存储方案

### 接口范围

1. `POST /users/intake/basic`
2. `GET /users/me`
3. `POST /events`
4. `POST /assets/upload-token`
5. `POST /users/intake/images`
6. `GET /jobs/{jobId}`
7. `GET /health`

### 交付物

1. 数据库迁移脚本
2. API 基础路由
3. 文件上传链路
4. 基础任务调度能力

### 验收标准

1. 用户资料可入库
2. 图片上传索引可入库
3. 事件可入库
4. API 基础资源可正常读写

---

## 7. Phase 2：画像 MVP 闭环

### 目标

打通最小核心业务闭环。

### 任务

1. 实现八字分析模块 V1
2. 实现问答模块 V1
3. 实现画像引擎 V1
4. 实现历史人物匹配 V1
5. 实现建议模块 V1
6. 实现成长档案基础时间线

### 技术任务

1. 建立八字规则计算逻辑
2. 定义标准 trait evidence 结构
3. 建立画像维度计算逻辑
4. 建立置信度与冲突检测逻辑
5. 建立历史人物基础样本库
6. 建立建议规则库 V1

### 接口范围

1. `GET /questionnaire/next`
2. `POST /questionnaire/answers`
3. `GET /profiles/current`
4. `GET /profiles/versions`
5. `POST /profiles/recompute`
6. `GET /matches/current`
7. `GET /advice/current`
8. `GET /archive/timeline`

### 交付物

1. 初始八字分析结果
2. 画像 V1 / V2
3. Top 3 历史人物匹配
4. 建议结果页所需数据

### 验收标准

1. 提交基础资料后能生成 V1 画像
2. 回答问答后能生成 V2 画像
3. 画像变化有日志可解释
4. 匹配结果与建议结果可生成

## 11. 首批实现优先级

如果从今天开始开发，建议严格按下面顺序执行：

1. Monorepo 初始化
2. FastAPI + PostgreSQL + Redis + Worker 启动
3. 数据表和迁移
4. 基础建档接口
5. 八字分析 V1
6. 问答接口
7. 画像引擎 V1
8. 匹配模块 V1
9. 建议模块 V1
10. Web 主流程页面
